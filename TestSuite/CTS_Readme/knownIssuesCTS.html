Additional Information

Known Issues
  • For security reasons Microsoft Internet Explorer requires that content that performs cross frame or cross-window scripting be loaded from the same domain ( http://msdn2.microsoft.com/en-us/library/ms533028.aspx). Because of this restriction, when testing a SCO in the SCO RTE Conformance Utility Test, the SCO that is being tested must be from the same domain as the domain for which the Test Suite is running. The SCOs use JavaScript to locate the LMS-provided API Instance, which requires the use of cross frame and/or cross-window scripting. For example, if the Test Suite is installed on your local machine on the C: drive, the SCO must be physically located on the C: drive also.

  • The Content Package Conformance Test is unable to test a content package using the http or ftp protocol.

  • When starting the LMS Conformance Test, check the Java Console. If there are any Java exceptions reported, close all browser windows and start the test again.

  • The links to the Detailed Logs contained in the Summary Logs will not be fully functional until the Test Suite completes the entire test.

  • Due to the design of the IMS Content Packaging Version 1.1.4 XML Schema Definition (XSD), an Internet connection may be required to perform the schema validation test in the Content Package Conformance Test.

  • Close the browser completely after each test before starting another Test Suite test.

  • If LMS Conformance Test is closed and reopened while having another internet browser open, the JVM may not be shut down and the socket may not be released. If this happens, the Applet that utilizes the socket throws an exception and fails to start. The following exception will be found in the Java Console if it encounters this issue:
    • SEVERE: Exception caught: java.net.BindException: Address already in use: JVM_Bind
      java.net.BindException: Address already in use: JVM_Bind
    Close all internet browsers, ensure that no JVMs are running (no Java icons are found in the task manager), and restart the Test Suite.

  • It is possible for a java.io.IOException to be thrown during the second activity of LMS Test Content Package DMI. The exception stems from issues with certain versions of the Java JRE and the joda.time plugin. The exception does not have any affect on conformance or the operation of the Test Suite.

  • When running the Test Suite Content Package Test or Manifest Test, the package will fail if the manifest or a schema listed in the manifest directly references a schema or sub-schema in a directory with non-English characters. For the purpose of this issue, a non-English character is any character outside of the Basic Latin Unicode code chart.

Environment Variable
  • An Environment Variable is set when the Test Suite is installed. If for any reason it would need manually reset, please follow these instructions.
    1. Right-click My Computer and select "Properties".
    2. Click the Advanced Tab.
    3. Click Environment Variables.
    4. Click New.
    5. Set the Variable name to SCORM4ED_TS111_HOME.
    6. Set the Variable value to <install_directory>.
    7. Click OK on the New Variable window.
    8. Click OK on the Environment Variables window.
    9. Click OK on the Systems Properties window.

Manifest Tester Utility
  • The Manifest Tester Utility allows the user to run the same Content Package validation checks as the Manifest Utility Test from a command line interface.
  • Instructions:
    • From the command prompt, navigate to the folder: <install_directory>\SCORM_2004_4th_Ed_TS_V1.1.1\lib
    • Enter the command java -jar adlvalidator.jar with the following possible parameters:
      • Filename
        • The path of a PIF(.zip) Content Package, the path of a non-PIF(.xml) Content Package manifest file, or the path of a directory containing PIF(.zip) Content Package files.
        • Multiple filenames can be provided to allow sequential validation of multiple Content Packages or Content Package manifests.
      • Application Profile Type
        • The given file(s) or folder contents can be validated against the Resource Application Profile if the option -r is entered after the file(s) or folder(s).
        • If no Application Profile Type option is provided, the Manifest Tester Utility, by default, will validate the entered file(s) and or folder contents against the Content Aggregation Application Profile.
      • Help
        • A brief summary of the Manifest Tester Utility instructions will be displayed at the command prompt if only the -h or -help options are entered.
  • Examples
    • java -jar adlvalidator.jar C:\MyFiles\MyContentPackage.zip
    • java -jar adlvalidator.jar C:\MyFiles\MyContentPackage1.zip C:\MyFiles\MyContentPackage2.zip
    • java -jar adlvalidator.jar C:\MyFiles\MyResourceContentPackage.zip -r
    • java -jar adlvalidator.jar C:\MyFiles\MyContentPackage.zip C:\MyFiles\nonPIF\imsmanifest.xml
    • java -jar adlvalidator.jar "C:\MyFiles\My ContentPackages\\"
