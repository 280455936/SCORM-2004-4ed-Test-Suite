Running the Content Package Conformance Test

To run the Content Package Conformance Test, click the corresponding link from the main Test Suite menu. A Java Plug-in Security Warning dialog box will appear since the Test Suite is reading and writing files from/to the local file system via a Java Applet. Click Grant this Session or Grant Always to allow the Test Suite applet to run. Clicking Deny will cause the Test Suite software to not function properly.

Content Package Overview

The Content Package Conformance Test verifies that the content package test subject complies with the SCORM 2004 4th Ed. Testing Requirements.

The purpose of this test is to verify that the Content Package test subject:

  • Contains an IMS Manifest at the root of the content package that:
    • Is named imsmanifest.xml.
    • Is well-formed.
    • Validates against the IMS Content Packaging XML Schema Definition (XSD) Version 1.1.4 ( imscp_v1p1.xsd found at http://www.imsglobal.org/ ).
    • Complies to the rules specified in the SCORM 2004 4th Ed. Content Package Application Profiles.
    • Uses extensions elements and if they are used, they are valid to their respective controlling document.
    • Verifies that the sequencing information is valid according to the IMS Simple Sequencing XML Schema Definition (XSD) ( imsss_v1p0.xsd and supporting files found at http://www.imsglobal.org/ ) if the IMS Manifest contains IMS Simple Sequencing Information.
    • If the IMS Manifest contains SCORM CAM Content Packaging extensions, verifies that the content packaging extensions are valid according to the SCORM 2004 4th Edition Content Packaging Extensions XML XSD Version 2.0 ( adlcp_v1p3.xsd found at http://www.adlnet.gov/ ).
    • Verifies that the sequencing extensions are valid according to the SCORM 2004 4th Ed. Sequencing Extensions XML XSD Version 2.0 ( adlseq_v1p3.xsd found at http://www.adlnet.gov/ ) if the IMS Manifest contains SCORM CAM Sequencing extensions.
    • Verifies that the navigation extensions are valid according to the SCORM 2004 4th Edition Navigation Extensions XML XSD Version 1.0 ( adlnav_v1p3.xsd found at http://www.adlnet.gov/ ) if the IMS Manifest contains SCORM CAM Navigation extensions.
  • Contains all supporting controlling documents required for XML Parsing at the root of the content package.
  • Contains optional Metadata XML Instances that are well-formed and valid according to the corresponding provided Controlling Document (e.g., XSD, DTD).
  • Contains conformant SCOs as determined by the SCO RTE Conformance Utility Test.
Operational Instructions
  1. Provide the Content Package Name, Content Package Version/Release Number and Content Package Vendor/Developer information.
  2. Click Continue.
  3. Select the type of test to be performed by selecting the appropriate radio button. You can test a Package Interchange File (PIF) in the form of a PKZIP 2.0.4 (or compatible), or an uncompressed Content Package (non-PIF).
  4. Click Continue.
  5. Select the type of SCORM Content Package Application Profile to be tested (i.e., Content Aggregation Content Package or Resource Content Package) by selecting the appropriate radio button.
  6. Click Continue.
  7. Click Browse to select the PIF zip file or the imsmanifest.xml file (for the non-PIF) that is to be tested.
  8. After clicking Begin Test the Test Suite will test the IMS Manifest and any referenced or inline metadata for schema validation only.
  9. Any SCOs or assets referenced by the IMS Manifest will be tested. If any SCOs or assets are listed in the IMS Manifest, the test will continue. Otherwise, testing is complete. If SCOs or assets exist:
    1. Enter a SCO Timeout Value (default is 20 seconds). This is the number of seconds the Test Suite will wait for the API method Initialize("") to be called. If the call is not made within this time, the Test Suite will end and the SCO will NOT be deemed conformant.
    2. Enter an "Asset Timeout Value" (default is 8 seconds). This is the amount of time the Test Suite will wait to verify that an Asset does not make any calls to the SCORM API.
    3. Click Continue.
    4. Optionally, you may enter initialization values for any of the following elements: Learner ID, Learner Name, Credit/No-Credit, Mode, or Launch Data. These values represent defaults typically provided by the LMS.
    5. Click Continue.
    6. Click Launch SCO to launch the first SCO or asset for testing.
    7. After the first SCO has completed testing, click Complete Test in the Instructions frame to quit that SCO test.
  • Any following SCO or asset will be launched and tested after you click OK on the pop-up window. Please note that the SCOs and assets are launched by the Content Package Test in a random order. All SCOs and assets listed in the manifest will be tested.
  • Use Abort Test if a call to Terminate("") is not received from the SCO being tested. Wait for a period of time to see if the call will be invoked. If it is not, clicking Abort Test will send a call to Terminate("") and end the testing, marking the SCO as non-conformant.

Operational Notes
Both Summary and Detailed Logs are automatically saved in the folder:
<install_directory>\SCORM_2004_4th_Ed_TS_V1.1.1\TestSuite\Logs\CP\<timestamp-test subject name>\

Note: Logs are saved in this folder as a result of a portability enhancement. If you need to move logs, move the entire folder and its contents since they are required to properly render the logs in the browser.
